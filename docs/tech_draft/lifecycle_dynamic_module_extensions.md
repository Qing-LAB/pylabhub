# Lifecycle Dynamic Module Extensions — Thread Awareness + User Data

**Status**: Draft (2026-03-31)
**Scope**: Extensions to the dynamic module system for thread-safe engine lifecycle
**Depends on**: Lifecycle system (done), ScriptEngine lifecycle integration (tech draft)

---

## 1. Motivation

Dynamic modules are loaded from arbitrary threads at runtime. The current API
lacks thread awareness and a safe mechanism to pass user context to callbacks.
This causes:

1. **No thread tracking**: The system doesn't know which thread initialized a
   module. If finalization runs on a different thread (e.g., Python's GIL),
   there is no detection or policy enforcement.

2. **Static globals for user data**: Callbacks are C function pointers
   (`void(*)(const char*)`). To access the owning object (e.g., a ScriptEngine),
   code resorts to file-static global pointers — unsafe for multi-instance
   scenarios and architecturally fragile.

3. **No pointer validity checking**: If the object behind a user-data pointer
   is destroyed before the lifecycle callback fires, the callback dereferences
   a dangling pointer.

## 2. Design

### 2.1 User data registration with generation key

```cpp
class LifecycleManager {
public:
    /// Validate function signature.
    /// @param userdata  The registered pointer.
    /// @param key       The generation key returned by register_user_data.
    /// @return true if the pointer is still valid and the key matches.
    using UserDataValidateFn = bool (*)(void *userdata, uint64_t key);

    /// Register a user-data pointer with a validation function.
    /// Returns a unique, monotonically increasing generation key.
    /// Lifecycle stores {ptr, key, validate_fn} internally.
    /// The caller should store the returned key on the pointed-to object
    /// so the validate function can confirm identity.
    [[nodiscard]] uint64_t register_user_data(void *ptr, UserDataValidateFn validate);

    /// Deregister user data by key. Called during cleanup.
    /// After this, any callback using this key will fail validation.
    void deregister_user_data(uint64_t key);
};
```

**Generation key lifecycle:**
- Generated by lifecycle: monotonic atomic counter (never reused).
- Returned to caller: stored on the user object (e.g., `engine->lifecycle_key_ = key`).
- Stored by lifecycle: in an internal map keyed by generation.
- On deregister: removed from lifecycle's map. Any future validate for this key fails.

### 2.2 ModuleDef extensions (dynamic modules only)

```cpp
class ModuleDef {
public:
    // ── NEW: User data ──────────────────────────────────────────────
    /// Associate this module with a registered user-data key.
    /// The userdata pointer is resolved from lifecycle's registry at callback time.
    void set_user_data_key(uint64_t key);

    // ── NEW: Thread awareness ───────────────────────────────────────
    /// Set whether finalize may run on a different thread than init.
    /// Default: true (any thread can finalize — backward compatible).
    /// Set to false for engines with thread affinity (e.g., Python GIL).
    void set_thread_safe_finalize(bool allow);

    // ── NEW: Callbacks with user data ───────────────────────────────
    /// Callback signature that receives the validated user-data pointer.
    using UserDataCallback = void (*)(void *userdata);

    void set_startup(UserDataCallback fn);
    void set_shutdown(UserDataCallback fn, std::chrono::milliseconds timeout);
};
```

**Backward compatibility:** The existing `LifecycleCallback` (`void(*)(const char*)`)
API remains unchanged. The new `UserDataCallback` is an alternative. Modules
use one or the other, not both. `InternalModuleDef` stores whichever was set.

### 2.3 InternalModuleDef additions

```cpp
struct InternalModuleDef {
    // Existing fields unchanged...

    // NEW:
    uint64_t user_data_key{0};            // 0 = no user data
    bool thread_safe_finalize{true};      // false = must finalize on init thread
    std::thread::id init_thread_id{};     // recorded by loadModuleInternal
};
```

### 2.4 loadModuleInternal changes

After the existing startup logic:

```cpp
// Record which thread performed initialization.
node.def.init_thread_id = std::this_thread::get_id();
```

### 2.5 processOneUnloadInThread changes

Before calling the shutdown callback:

```cpp
// Thread awareness check.
if (!def.thread_safe_finalize &&
    def.init_thread_id != std::thread::id{} &&
    std::this_thread::get_id() != def.init_thread_id)
{
    LOGGER_WARN("[lifecycle] module '{}' requires finalize on init thread "
                "(init={}, current={}) — skipping shutdown callback",
                name, def.init_thread_id, std::this_thread::get_id());
    // Module marked as contaminated (not cleanly shut down).
    return;
}

// User data validation.
if (def.user_data_key != 0)
{
    auto [ptr, validate] = resolve_user_data(def.user_data_key);
    if (!validate || !validate(ptr, def.user_data_key))
    {
        LOGGER_WARN("[lifecycle] module '{}' user data validation failed — "
                    "skipping shutdown callback", name);
        return;
    }
    // Call shutdown with validated userdata.
    def.shutdown_with_userdata(ptr);
}
else
{
    // Legacy path: call shutdown(const char *arg) as before.
    def.shutdown();
}
```

### 2.6 Validation flow

```
Engine construction:
    engine = new NativeEngine();

Registration:
    key = LifecycleManager::register_user_data(engine, engine_validate_fn);
    engine->lifecycle_key_ = key;          // engine stores its own key

Module setup:
    ModuleDef mod("ScriptEngine:native:libfoo.so:PROD-A");
    mod.set_user_data_key(key);
    mod.set_thread_safe_finalize(true);    // native is thread-safe
    mod.set_startup(engine_startup);       // receives engine* as userdata
    mod.set_shutdown(engine_shutdown, 5s); // receives engine* as userdata

LoadModule (on worker thread):
    init_thread_id = this_thread
    validate(engine, key) → engine->lifecycle_key_ == key → true
    startup(engine) → engine->initialize()

UnloadModule (async, on shutdown thread):
    check thread_safe_finalize → true (native) → proceed
    validate(engine, key) → engine->lifecycle_key_ == key → true
    shutdown(engine) → engine->finalize()

If engine already destroyed (abnormal path):
    validate(engine, key) → engine->lifecycle_key_ != key (memory recycled) → false
    → skip callback, log warning
```

### 2.7 Validate function contract

```cpp
// Provided by the user (e.g., ScriptEngine base class):
static bool engine_validate(void *userdata, uint64_t key) {
    auto *engine = static_cast<ScriptEngine*>(userdata);
    // Check: is this memory still a valid engine with the right key?
    return engine != nullptr && engine->lifecycle_key_ == key;
}
```

The validate function is the user's guarantee that the pointer is still valid.
The generation key prevents false positives from memory reuse — the new occupant
of the same address will not have the matching key at the `lifecycle_key_` offset.

## 3. ScriptEngine integration

### 3.1 Base class additions

```cpp
class ScriptEngine {
public:
    // Existing interface unchanged...

    /// Create a lifecycle module for this engine.
    /// Called by the role host from worker_main_().
    ModuleDef make_lifecycle_module();

    /// Module name for lifecycle registration.
    virtual std::string lifecycle_module_name() const = 0;

protected:
    uint64_t lifecycle_key_{0};  // set during make_lifecycle_module

    // Existing virtual hooks unchanged:
    // init_engine_(), build_api_(), finalize_engine_()
};
```

### 3.2 Per-engine overrides

| Engine | `lifecycle_module_name()` | `set_thread_safe_finalize` |
|--------|--------------------------|---------------------------|
| Python | `"ScriptEngine:python"` | `false` (GIL thread affinity) |
| Lua | `"ScriptEngine:lua:<uid>"` | `true` |
| Native | `"ScriptEngine:native:<lib>:<uid>"` | `true` |

### 3.3 Role host usage

```cpp
worker_main_():
    auto mod = engine_->make_lifecycle_module();
    RegisterDynamicModule(std::move(mod));
    LoadModule(engine_->lifecycle_module_name());  // → initialize on worker thread

    engine_->load_script(...);
    setup_infrastructure_();
    engine_->build_api(ctx);
    // ... run ...
    engine_->invoke_on_stop();

    engine_->finalize();                           // direct, on worker thread
    UnloadModule(engine_->lifecycle_module_name()); // lifecycle bookkeeping
    // wait_for_unload not needed: finalize already done, shutdown callback is no-op
    teardown_infrastructure_();
```

## 4. Changes Required

### 4.1 Lifecycle system (src/include/utils/, src/utils/service/)

- `lifecycle.hpp`: Add `register_user_data`, `deregister_user_data` to LifecycleManager
- `module_def.hpp`: Add `set_user_data_key`, `set_thread_safe_finalize`, `UserDataCallback`
  overloads for `set_startup`/`set_shutdown`
- `lifecycle_impl.hpp`: Add `user_data_registry_` (map<uint64_t, {ptr, validate_fn}>),
  `next_generation_key_` atomic counter
- `lifecycle_dynamic.cpp`: Thread-ID recording in `loadModuleInternal`,
  thread check + validation in `processOneUnloadInThread`

### 4.2 ScriptEngine (src/include/utils/script_engine.hpp)

- Add `lifecycle_key_` member, `make_lifecycle_module()`, `lifecycle_module_name()`
- Add `EngineState` enum, `state_` member (tracked by initialize/load_script/build_api/finalize)
- Static `engine_validate` function

### 4.3 Each engine subclass

- Override `lifecycle_module_name()`
- No other changes (existing init_engine_/build_api_/finalize_engine_ hooks unchanged)

### 4.4 Role host worker_main_() (all three)

- Add RegisterDynamicModule + LoadModule at start
- Add UnloadModule after finalize
- Remove direct initialize() call (done by LoadModule startup callback)

### 4.5 main() (all three)

- Remove if/else engine creation (move to factory)
- Engine created before role host, lifecycle module registered by role host

---

## 5. What Does NOT Change

- Static module lifecycle (Logger, ZMQ, Crypto, etc.)
- Existing LifecycleCallback API (backward compatible)
- ScriptEngine invoke interface (invoke_produce, etc.)
- L2 tests (use engines directly without lifecycle)
- LifecycleGuard constructor/destructor behavior
