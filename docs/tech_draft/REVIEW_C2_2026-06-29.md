    independently pinned at one layer.

### F7 — `apply_5b_canonical_fields` keys off payload uid, not wire identity

- **File:** `tests/test_layer3_datahub/workers/datahub_broker_workers.cpp:245`
  (decoration function); broker side `broker_service.cpp` §I10
  enforcement code.
- **Symptom:** Decoration reads `zmq_pubkey` from KeyStore[payload.role_uid]
  while raw_req CURVE-authenticates as `role_identity_name`.  When
  they diverge (e.g. `broker_gate_consumer_reg_req_rejects_producer_tag`:
  CURVE = cregwt, payload = intruder), decoration writes intruder's
  pubkey.  Test currently passes only because the role-tag gate
  fires before any (zap_user_id, zmq_pubkey) cross-check.
- **Root cause (suspected — verify):** **Both halves are buggy.**
  - Test: decoration should key off wire identity so the payload
    reflects who is actually CURVE-authenticated.
  - Production: HEP-CORE-0036 §I10 says broker should enforce that
    the ZAP-authenticated peer pubkey matches the payload's
    declared role_uid → known_roles binding.  If that check is
    missing, an attacker who has any seeded identity (even a
    different role) could send REG_REQ payloads under arbitrary
    role_uid values.  This is a real security gap.
- **Verification needed:** Read `verify_known_role_binding` /
  REG_REQ handler in `broker_service.cpp` and HEP-CORE-0036 §I10.
  If broker does enforce, then test fix only.  If not, that's a
  production bug to file + fix.
