## 1. What shipped

A new optional script callback:

```python
def on_channel_closing(channel, reason, api):
    # channel: str ÔÇö name of the just-closed channel
    # reason:  str ÔÇö broker-side reason string (e.g. "producer_deregistered",
    #               "pending_timeout", "script_requested")
    # api:     RoleAPIBase ÔÇö same proxy the other callbacks receive
    ...
```

Fires when a `CHANNEL_CLOSING_NOTIFY` arrives in the per-cycle
`msgs` list AND the script defines the callback.  When dispatched,
the entry is removed from `msgs` before `on_produce`/`on_consume`
runs (single-delivery).  When NOT defined, the notify stays in
`msgs` ÔÇö existing scripts that scan for it keep working.

The framework takes no action on its own.  The script decides:
call `api.stop()` to exit; mutate role-instance state for a later
cycle to act on (retry, re-register, log, etc.); or do nothing.

## 2. Why this exists

Pre-M1.2 there was a two-step teardown: `CHANNEL_CLOSING_NOTIFY`
(informational, grace timer running) Ôćĺ `FORCE_SHUTDOWN`.  M1.2
removed `FORCE_SHUTDOWN` and the grace window; channel close
became atomic ÔÇö when the last producer-presence reaches
`Disconnected`, the broker removes the `ChannelEntry` and fans out
one `CHANNEL_CLOSING_NOTIFY` in the same handler.

The notify is now post-fact ("the channel is gone right now"),
not informational.  Scripts that care need a clear hook.  M1.5
adds it as a dedicated callback; M1.2 already delivered the notify
via `msgs` (that path stays).

## 4. Contract ÔÇö one paragraph

When `CHANNEL_CLOSING_NOTIFY` arrives in `msgs`:
1. If `engine.has_callback("on_channel_closing")` is true: framework
   calls `engine.invoke_on_channel_closing(channel, reason)` once
   per entry and removes those entries from `msgs`.
2. If false: framework leaves `msgs` unchanged; the script's
   `on_produce`/`on_consume` receives the entries and may scan.

No new stop machinery.  No auto-stop default.  No `StopReason`
addition.  Script uses existing `api.stop()` if it wants to quit.

## 5. Tests

`tests/test_layer2_service/test_dispatch_channel_closing.cpp`
(Pattern 1+, Logger-only guard).  6 tests cover:
- No callback Ôćĺ msgs unchanged, engine not called.
- Callback present, single notify Ôćĺ dispatched + removed.
- Callback present, mixed entries Ôćĺ only notifies removed, others
  preserved in original order.
- Empty msgs Ôćĺ no-op, regardless of callback state.
- Notify with missing fields Ôćĺ empty-string args, no crash.

## 7. What the original 2026-05-12 draft proposed but did NOT ship

Recorded for historical traceability ÔÇö these are the design-phase
decisions that got rejected in the implementation discussion:

| Proposed knob | Why it was rejected |
|---|---|
| `auto_stop_on_channel_close` config flag (per-role) | Framework should not encode policy; script decides. |
| `StopReason::ChannelClosed` new enum value | No new stop machinery needed; existing `api.stop()` suffices.  Script can log + call existing stop. |
| Per-role-type defaults (D1c: consumer=true, producer=false, processor=true) | Same reason ÔÇö script decides, not framework. |
| Auto-stop scope decision (D2: any-channel vs all-inputs) | Framework doesn't auto-stop, so the scope question doesn't apply. |
| Return-value protocol from the callback | Script uses `api.stop()` or instance state ÔÇö no protocol needed. |

Final landed shape is just "deliver the callback if defined; remove
from msgs to avoid dual delivery; otherwise do nothing."
