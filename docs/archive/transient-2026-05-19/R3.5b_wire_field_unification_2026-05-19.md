# R3.5b — Wire-Field Unification + Grammar + Side-Aware Tag

**Status**: ✅ SHIPPED 2026-05-19 (audit-track R3.5b)
**Tests**: 1951/1951 pass (1945 baseline + 6 new gate-validation L3 tests).
**Scope**: broker_proto 4→5 (clean cut, mirroring B1 channel→band rename precedent)
**Triggered by**: User-flagged bug — broker silently accepts empty `consumer_uid` on CONSUMER_REG_REQ; role-presence then silent-drops, breaking heartbeat keying.

## Why

Three independent issues entwined at the wire boundary:

1. **Empty uid silent-drop bug.** Default `RoleIdentityPolicy::Open` accepts empty uid; `_on_consumer_joined` then skips `upsert_role_locked` (gated on `!consumer_uid.empty()`), and `_on_heartbeat` silent-returns when `roles.find(role_uid)` misses. Net effect: no warning, broker_state silently incomplete.

2. **Wire-field naming divergence.** `role_uid` (REG, DEREG, ROLE_INFO, ROLE_PRESENCE, BAND_*), `consumer_uid`/`consumer_name` (CONSUMER_REG), `uid` (HEARTBEAT), `sender_uid` (BAND_BROADCAST + IncomingMessage). Per HEP-CORE-0033 §G2.2.0b the role tag (`prod`/`cons`/`proc`) is embedded inside the uid value; consumer-specific field names are redundant.

3. **Side-aware semantics absent.** Grammar validator accepts any `(prod|cons|proc).x.y` for any handler, so `role_uid="prod.foo.bar"` would pass on CONSUMER_REG_REQ (semantically nonsensical).

Per-side tag policy accounting for processor dual-role:

| Handler | Expected tag set |
|---|---|
| REG_REQ, DEREG_REQ | `{prod, proc}` |
| CONSUMER_REG_REQ, CONSUMER_DEREG_REQ | `{cons, proc}` |
| HEARTBEAT_REQ | derived from `role_type` field |
| ROLE_PRESENCE_REQ, ROLE_INFO_REQ | `{prod, cons, proc}` |
| BAND_JOIN_REQ, BAND_LEAVE_REQ, BAND_BROADCAST_REQ | `{prod, cons, proc}` |

## Wire format — proto 5

After this patch all gates accept the canonical pair `role_uid` + `role_name`. HEARTBEAT carries `role_uid` + `role_type`. Notify payloads use `role_uid` instead of `consumer_uid` / `sender_uid`.

| Wire message | uid field | name field | type field |
|---|---|---|---|
| `REG_REQ` | `role_uid` | `role_name` | — |
| `CONSUMER_REG_REQ` | `role_uid` (was `consumer_uid`) | `role_name` (was `consumer_name`) | — |
| `DEREG_REQ` | `role_uid` | — | — |
| `CONSUMER_DEREG_REQ` | `role_uid` | — | — |
| `HEARTBEAT_REQ` | `role_uid` (was `uid`) | — | `role_type` |
| `ROLE_PRESENCE_REQ` / `ROLE_INFO_REQ` | `role_uid` | — | — |
| `BAND_JOIN_REQ` / `BAND_LEAVE_REQ` | `role_uid` | `role_name` | — |
| `BAND_BROADCAST_REQ` | `role_uid` (was `sender_uid`) | — | — |
| `CONSUMER_DIED_NOTIFY` body | `role_uid` (was `consumer_uid`) | — | — |
| `BAND_BROADCAST_NOTIFY` body | `role_uid` (was `sender_uid`) | — | — |
| `IncomingMessage.sender_uid` | `role_uid` (struct field rename) | — | — |
| `PyInboxMsg.sender_uid` | `role_uid` (Python attribute rename) | — | — |

## Validator helper signature

```cpp
[[nodiscard]] std::optional<nlohmann::json>
validate_identity_fields(const std::string& channel_name,
                         const std::string& role_uid,
                         const std::string& role_name,
                         std::initializer_list<std::string_view> expected_tags,
                         const std::string& corr_id,
                         const char*        handler_label) const;

[[nodiscard]] std::optional<nlohmann::json>
validate_role_uid_only(const std::string& role_uid,
                       std::initializer_list<std::string_view> expected_tags,
                       const std::string& corr_id,
                       const char*        handler_label) const;
```

Returns `INVALID_REQUEST` on grammar failure; `INVALID_ROLE_TAG` on tag-mismatch.

## Migration steps (sequence)

1. Extend validator (grammar + tag-set + uses `extract_role_tag`).
2. Rewire broker handlers to read canonical fields + apply validator with per-handler tag policy.
3. Update broker senders to emit canonical field names on notify payloads.
4. Update role-side `BrokerRequestComm::send_*` to populate canonical fields.
5. Update `role_reg_payload.hpp` builder (`consumer_uid` → `role_uid`).
6. Update `cycle_ops.hpp` reader.
7. Update `IncomingMessage` struct + `PyInboxMsg` + Python bindings + Lua bindings.
8. Bump `plh_version_registry.hpp` broker_proto 4→5.
9. Update tests that hand-build wire payloads.
10. Update HEP-CORE-0023 §2.5 + HEP-CORE-0033 §G2.2 wire-field tables.
