## 0. Correction (audit D1, 2026-05-18)

The initial M1.5 ship (commit `c177c99`, 2026-05-14) implemented
**no-default-action** semantics: the framework only fired the
callback if defined; otherwise the notify stayed in `msgs` for
generic scan, and the role did NOT stop.

This was the OPPOSITE of the user's design decision locked on
2026-05-15T03:29:14 (the same day as the M1.5 ship, but a few hours
later — the design discussion happened after the initial code
landed; the code was not corrected at the time):

> "on_channel_closing, if the user provides it, will replace the
>  call of api.stop().  the user can call this within its own
>  script, or decide to keep the loop running (but can manipulate
>  some internal state like book keeping and the next iteration
>  can sense this and perform additional operations like
>  re-register or something else).  The user can decide to keep
>  the role going like a zombie if he/she decides to.  this is
>  really just a callback that replaces the default straightforward
>  stop()"

And clarified at 2026-05-15T03:30:03:

> "we don't need another stop mechanism.  this is just a message
>  that will lead to stop by default"

The canonical semantics — implemented under audit D1 (2026-05-18)
via the unified `kNotificationTable[]` in `cycle_ops.hpp` with
`invoke_user_channel_closing` (user override) + `default_channel_closing`
(native default):

  - **If the script defines `on_channel_closing`**, the dispatcher
    calls `engine.invoke_on_channel_closing(channel, reason)` and
    the notify is consumed from `msgs`.  Framework's default does
    not fire.
  - **If NOT defined**, the dispatcher calls
    `default_channel_closing(msg, StopRequestor{core})` which sets
    `StopReason::ChannelClosed` and calls `core.request_stop()`.
    The notify is still consumed from `msgs` (no fall-through to
    generic scan).

See HEP-CORE-0011 §"Notification dispatch" for the full uniform
model (replaces the earlier Pattern A/B taxonomy with one rule:
every known notification has BOTH a user-override hook AND a
native default; the override REPLACES the default).

This brings `on_channel_closing` into uniform Pattern-A dispatch
with the new `on_hub_dead` callback (audit D2) — see HEP-CORE-0011
§"Notification dispatch" and the per-handler docstring in
`cycle_ops.hpp`.

The rest of this document describes the INITIAL design and is
preserved for traceability.  **The "framework takes no action on
its own" line in §1 is INCORRECT under the corrected semantics.**

## 1. What shipped (initial — superseded by §0)

A new optional script callback:

```python
def on_channel_closing(channel, reason, api):
    # channel: str — name of the just-closed channel
    # reason:  str — broker-side reason string (e.g. "producer_deregistered",
    #               "pending_timeout", "script_requested")
    # api:     RoleAPIBase — same proxy the other callbacks receive
    ...
```

Fires when a `CHANNEL_CLOSING_NOTIFY` arrives in the per-cycle
`msgs` list AND the script defines the callback.  When dispatched,
the entry is removed from `msgs` before `on_produce`/`on_consume`
runs (single-delivery).  When NOT defined, the notify stays in
`msgs` — existing scripts that scan for it keep working.

The framework takes no action on its own.  The script decides:
call `api.stop()` to exit; mutate role-instance state for a later
cycle to act on (retry, re-register, log, etc.); or do nothing.

**See §0 — this paragraph describes the INITIAL behavior, which
was reversed under audit D1 (2026-05-18).  Default is now "framework
calls `core.request_stop()` with `StopReason::ChannelClosed`" when
no callback is defined.**

## 2. Why this exists

Pre-M1.2 there was a two-step teardown: `CHANNEL_CLOSING_NOTIFY`
(informational, grace timer running) → `FORCE_SHUTDOWN`.  M1.2
removed `FORCE_SHUTDOWN` and the grace window; channel close
became atomic — when the last producer-presence reaches
`Disconnected`, the broker removes the `ChannelEntry` and fans out
one `CHANNEL_CLOSING_NOTIFY` in the same handler.

The notify is now post-fact ("the channel is gone right now"),
not informational.  Scripts that care need a clear hook.  M1.5
adds it as a dedicated callback; M1.2 already delivered the notify
via `msgs` (that path stays).

| Layer | File | Purpose |
|---|---|---|

| Python impl | `src/scripting/python_engine.{hpp,cpp}` | Caches `py_on_channel_closing_` at load, invokes with `(channel, reason, api_obj_)` under GIL |
| Lua impl | `src/scripting/lua_engine.{hpp,cpp}` | Caches `ref_on_channel_closing_` registry ref, `pcall` with `(channel, reason, api)` |
| Native impl | `src/include/utils/native_engine.hpp` + `src/utils/service/native_engine.cpp` | Resolves `on_channel_closing` symbol, calls with JSON-encoded args |
| Dispatch site | `src/utils/service/cycle_ops.hpp` (`dispatch_channel_closing` helper) | Called from each Ops's `invoke_and_commit` before the data callback; removes matching entries from `msgs` |

## 4. Contract — one paragraph

When `CHANNEL_CLOSING_NOTIFY` arrives in `msgs`:
1. If `engine.has_callback("on_channel_closing")` is true: framework
   calls `engine.invoke_on_channel_closing(channel, reason)` once
   per entry and removes those entries from `msgs`.
2. If false: framework leaves `msgs` unchanged; the script's
   `on_produce`/`on_consume` receives the entries and may scan.

No new stop machinery.  No auto-stop default.  No `StopReason`
addition.  Script uses existing `api.stop()` if it wants to quit.

## 5. Tests

`tests/test_layer2_service/test_dispatch_channel_closing.cpp`
(Pattern 1+, Logger-only guard).  6 tests cover:
- No callback → msgs unchanged, engine not called.
- Callback present, single notify → dispatched + removed.
- Callback present, mixed entries → only notifies removed, others
  preserved in original order.
- Empty msgs → no-op, regardless of callback state.
- Notify with missing fields → empty-string args, no crash.

## 7. What the original 2026-05-12 draft proposed but did NOT ship

Recorded for historical traceability — these are the design-phase
decisions that got rejected in the implementation discussion:

| Proposed knob | Why it was rejected |
|---|---|
| `auto_stop_on_channel_close` config flag (per-role) | Framework should not encode policy; script decides. |
| `StopReason::ChannelClosed` new enum value | No new stop machinery needed; existing `api.stop()` suffices.  Script can log + call existing stop. |
| Per-role-type defaults (D1c: consumer=true, producer=false, processor=true) | Same reason — script decides, not framework. |
| Auto-stop scope decision (D2: any-channel vs all-inputs) | Framework doesn't auto-stop, so the scope question doesn't apply. |
| Return-value protocol from the callback | Script uses `api.stop()` or instance state — no protocol needed. |

Final landed shape is just "deliver the callback if defined; remove
from msgs to avoid dual delivery; otherwise do nothing."
