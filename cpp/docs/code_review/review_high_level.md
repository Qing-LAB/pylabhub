Subdirectory Notes

cpp/src/utils/shm, cpp/src/utils/ipc, cpp/src/utils/hub: strongest part of the tree. The code is large but the L3 suite around broker protocol, queues, processor core, schema, recovery, and transport switching is broad, especially test_datahub_hub_processor.cpp (line 19).
cpp/src/producer, cpp/src/consumer, cpp/src/processor: config parsing and CLI surfaces are tested reasonably well, but the real script-host runtime behavior is under-tested. The biggest gaps are generated-template validity, runtime timing semantics, startup coordination, and Windows behavior.
cpp/src/scripting: this layer is structurally important and lightly exercised. Base ScriptHost behavior has tests, but PythonRoleHostBase, RoleHostCore, and role_main_helpers have little or no direct coverage.
cpp/src/hub_python: weakest direct test coverage. AdminShell gets integration coverage, but HubScript, HubScriptAPI, PythonInterpreter, and the pybind namespace wiring do not have the same depth of focused tests.
cpp/src/utils/service, cpp/src/utils/core, cpp/src/utils/logging, cpp/src/utils/schema: generally solid and well-covered, but the review surfaced doc/security drift around hub config/vault handling and Windows prompt behavior rather than core algorithmic faults.
Tests

Overall quality is good for the DataHub core: the layered layout is sensible, there is substantial multi-process testing, and the L3 suite covers many protocol and concurrency paths.
Overall completeness is weaker at the binary/script-host boundary. Missing or insufficiently protected areas are: --init output round-tripping, processor loop timing runtime behavior, startup coordination, hub secret placement, Windows non-interactive flows, Windows password masking, and direct hub_python/scripting behavior.
There is also a correctness/documentation problem in the test guidance itself: cpp/docs/README/README_testing.md (line 121) still tells developers to build and run targets that no longer exist.
